# llama.cpp ROCm Setup

Run local LLMs with proper **tool calling support** using llama.cpp with AMD GPU acceleration.

## Why llama.cpp instead of Ollama?

Ollama's OpenAI-compatible API (`/v1` endpoints) doesn't properly translate tool calls for many local models. This causes issues with AI coding assistants like OpenCode - models output malformed XML-style tags instead of proper JSON tool calls.

**llama.cpp's native server handles tool calling correctly**, making it the better choice for agentic workflows.

Both can run side-by-side:
- Ollama on port 11434 (for simple chat/generation)
- llama.cpp on port 8080 (for tool calling with OpenCode)

## Quick Start

```bash
# Run the setup script
./setup.sh

# Start the server with your chosen model
./start-server.sh qwen3

# In another terminal, run OpenCode
cd /your/project
opencode
# Use '/models' to select llama.cpp provider
```

## Requirements

**Required:**
- Linux (ROCm/HIP is Linux-only)
- AMD GPU with ROCm support (RX 6000/7000 series, Radeon VII, etc.)
- ROCm/HIP installed (`rocminfo` and `hipcc` available)
- CMake, make/ninja, git

**Optional but recommended:**
- `gum` - for interactive model selection menu
- `huggingface-cli` - for resumable downloads
- `jq` - for JSON config manipulation

## Scripts

| Script | Description |
|--------|-------------|
| `setup.sh` | Main setup - builds llama.cpp, downloads models, generates OpenCode config |
| `start-server.sh` | Start llama-server with a model |
| `download-model.sh` | Download individual models from HuggingFace |
| `sync-opencode-config.sh` | Sync downloaded models to OpenCode configuration |
| `uninstall.sh` | Clean removal with interactive selection |

## Configuration Files

| File | Description |
|------|-------------|
| `models.conf` | Model definitions (HuggingFace repos, filenames, sizes) |
| `models-metadata.conf` | Display names and context limits for OpenCode |
| `.env` | Local configuration (generated by setup.sh) |
| `.env.example` | Template with all available settings |

### Environment Variables

The `.env` file controls server and GPU settings. Key variables:

| Variable | Default | Description |
|----------|---------|-------------|
| `LLAMA_PORT` | 8080 | Server port |
| `LLAMA_CONTEXT` | 32768 | Context window size (tokens) |
| `LLAMA_HOST` | 127.0.0.1 | Host to bind to |
| `GPU_LAYERS` | -1 | GPU layers (-1 = all, 0 = CPU only) |
| `LLAMA_PARALLEL` | 2 | Parallel request slots |
| `LLAMA_BATCH_SIZE` | 512 | Batch size for prompt processing |
| `LLAMA_THREADS` | auto | CPU threads for inference |
| `GPU_TARGET` | auto | GPU architecture (e.g., gfx1100) |
| `HSA_OVERRIDE_GFX_VERSION` | auto | HSA version for ROCm compatibility |
| `LLAMA_FLASH_ATTENTION` | 1 | Enable flash attention (1=on, 0=off) |
| `LLAMA_MMAP` | 1 | Memory-mapped model loading |
| `LLAMA_MLOCK` | 0 | Memory locking (prevents swapping) |
| `LLAMA_CPP_DIR` | ./llama.cpp | Path to llama.cpp repository |
| `MODELS_DIR` | ./models | Path to GGUF model files |

See `.env.example` for detailed documentation of each setting.

## Usage

### Setup

```bash
# Full interactive setup
./setup.sh

# Non-interactive with defaults
./setup.sh --non-interactive

# Skip building (use existing build)
./setup.sh --skip-build

# Force rebuild
./setup.sh --force-rebuild

# Check current status
./setup.sh --status

# Update llama.cpp to latest version
./setup.sh --update

# Fix GPU permissions
./setup.sh --fix-permissions

# Verify downloaded models
./setup.sh --verify
```

### Starting the Server

```bash
# Start with a model ID from models.conf
./start-server.sh qwen3-30b-q4km

# Custom port and context size
./start-server.sh -p 8081 -c 65536 qwen3-30b-q4km

# List available models
./start-server.sh --list

# Run benchmark after starting
./start-server.sh --benchmark qwen3-8b-q4km

# All options
./start-server.sh --help
```

### Downloading Models

```bash
# List all available models
./download-model.sh --list

# Download a specific model
./download-model.sh qwen3-30b-q4km

# Show model info
./download-model.sh --info qwen3-8b-q4km

# Re-download (force)
./download-model.sh --force qwen3-30b-q4km

# Search HuggingFace for models
./download-model.sh --search "qwen gguf"
```

### Uninstalling

```bash
# Interactive removal
./uninstall.sh

# Remove everything
./uninstall.sh --all

# Remove specific components
./uninstall.sh --build          # Just the llama.cpp build
./uninstall.sh --models         # Just downloaded models
./uninstall.sh --config         # Just local .env
./uninstall.sh --opencode       # Remove from OpenCode config

# Dry run (show what would be removed)
./uninstall.sh --dry-run
```

## Models

Models are defined in `models.conf`. Edit this file to add custom GGUF models.

### Recommended Models by VRAM

| VRAM | Recommended Models |
|------|-------------------|
| 8GB | `qwen3-8b-q4km` (5GB), `qwen2.5-coder-3b-q4km` (2GB) |
| 12GB | `qwen3-14b-q4km` (9GB), `qwen2.5-coder-14b-q4km` (9GB) |
| 16GB | `qwen3-32b-q3km` (16GB), `deepseek-r1-14b-q4km` (9GB) |
| 24GB | `qwen3-30b-q4km` (18GB), `qwen3-32b-q4km` (20GB), `qwen2.5-coder-32b-q4km` (20GB) |

### Model Categories

| Category | Description | Default Context |
|----------|-------------|-----------------|
| `small` | Test models, quick verification | 8192 |
| `autocomplete` | IDE code completion | 32768 |
| `general` | All-purpose chat and coding | 32768 |
| `reasoning` | Models that show thinking process | 32768 |
| `coding` | Optimized for code generation | 65536 |

### Adding Custom Models

Edit `models.conf`:

```
# FORMAT: category|model_id|huggingface_repo|gguf_filename|size|description
coding|my-model-q4km|user/repo-GGUF|model-q4_k_m.gguf|10GB|My custom model
```

Then add metadata in `models-metadata.conf`:

```
# FORMAT: model_id|display_name|context_limit|output_limit
my-model-q4km|My Custom Model Q4_K_M|65536|16384
```

Find GGUF models at: https://huggingface.co/models?sort=trending&search=gguf

## OpenCode Integration

The setup script automatically generates an OpenCode configuration at:
`~/.config/opencode/opencode.json`

If you already have a config, use `sync-opencode-config.sh` to merge:

```bash
# Full sync (updates llama.cpp provider, keeps other providers)
./sync-opencode-config.sh

# Dry run to see what would change
./sync-opencode-config.sh --dry-run

# Restore from backup
./sync-opencode-config.sh --restore
```

### Manual Configuration

Add to your `opencode.json`:

```json
{
  "provider": {
    "llama.cpp": {
      "npm": "@ai-sdk/openai-compatible",
      "name": "llama.cpp (local)",
      "options": {
        "baseURL": "http://127.0.0.1:8080/v1"
      },
      "models": {
        "qwen3-30b-q4km": {
          "name": "Qwen3 30B MoE Q4_K_M (Agentic)",
          "tools": true,
          "limit": { "context": 65536, "output": 16384 }
        }
      }
    }
  }
}
```

## Performance Tuning

### Context Size vs VRAM

Context size significantly impacts VRAM usage. Use smaller context if you're running out of memory:

```bash
# Lower context for limited VRAM
./start-server.sh -c 8192 qwen3-32b-q4km

# Higher context for complex tasks (needs more VRAM)
./start-server.sh -c 65536 qwen3-30b-q4km
```

Approximate VRAM usage for a 32B model:
- 8K context: ~20GB
- 16K context: ~21GB
- 32K context: ~23GB
- 64K context: ~27GB

### Batch Size and Parallelism

Edit `.env` or use command line flags:

```bash
# Increase batch size for faster prompt processing
./start-server.sh -b 1024 qwen3-30b-q4km

# Multiple parallel requests (needs more VRAM)
./start-server.sh --parallel 4 qwen3-8b-q4km
```

### Partial GPU Offload

If a model doesn't fit entirely in VRAM, use partial offload:

```bash
# Offload only 30 layers to GPU, rest on CPU
./start-server.sh -n 30 qwen3-32b-q4km
```

### Thread Count

By default, llama.cpp uses all available cores. Override with:

```bash
./start-server.sh -t 8 qwen3-30b-q4km  # Use 8 threads
```

## Troubleshooting

### Build fails with HIP errors

1. Ensure ROCm is properly installed:
```bash
rocminfo          # Should list your GPU
hipconfig --version
```

2. Check HIP compiler:
```bash
hipcc --version
```

3. If newly installed, reboot or reload modules:
```bash
sudo modprobe amdgpu
```

### Permission denied accessing GPU

Fix group permissions:
```bash
./setup.sh --fix-permissions
# Then log out and back in
```

Or manually:
```bash
sudo usermod -aG video $USER
sudo usermod -aG render $USER
# Log out and back in
```

### Server won't start

1. Check the model file exists and is valid:
```bash
./setup.sh --verify
./start-server.sh --list
```

2. Verify GPU settings:
```bash
./setup.sh --status
```

3. Check ROCm runtime with HSA override:
```bash
HSA_OVERRIDE_GFX_VERSION=11.0.0 rocminfo
```

4. Try with verbose output:
```bash
LLAMA_LOG_LEVEL=debug ./start-server.sh qwen3-8b-q4km
```

### Tool calls not working in OpenCode

1. Ensure you're using the llama.cpp provider (not Ollama):
   - Use `/models` in OpenCode to switch

2. Check server is running:
```bash
curl http://127.0.0.1:8080/health
```

3. Verify model supports tool calling:
   - Qwen3 models: recommended
   - Qwen2.5-Coder: works well
   - DeepSeek-R1: may have issues with tool calls

### Out of VRAM (OOM errors)

Options in order of preference:

1. **Use a smaller model:**
```bash
./start-server.sh qwen3-8b-q4km  # Only ~5GB
```

2. **Reduce context size:**
```bash
./start-server.sh -c 8192 qwen3-32b-q4km
```

3. **Reduce batch size:**
```bash
./start-server.sh -b 256 qwen3-32b-q4km
```

4. **Use partial GPU offload:**
```bash
./start-server.sh -n 20 qwen3-32b-q4km  # Only 20 layers on GPU
```

5. **Use smaller quantization:**
```bash
# Use Q3_K_M instead of Q4_K_M (smaller but lower quality)
./download-model.sh qwen3-32b-q3km
```

### Model corrupted or incomplete

Verify and re-download:
```bash
./setup.sh --verify
./download-model.sh --force qwen3-30b-q4km
```

### Slow inference

1. Ensure GPU is being used (check `rocm-smi` while running)

2. Check HSA_OVERRIDE_GFX_VERSION is set correctly:
```bash
cat .env | grep HSA
```

3. Enable flash attention (should be default):
```bash
./start-server.sh --flash-attn qwen3-30b-q4km
```

4. Increase batch size for longer prompts:
```bash
./start-server.sh -b 1024 qwen3-30b-q4km
```
